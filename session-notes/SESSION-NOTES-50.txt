DESIGN ARC — how we got from "mtime gating" to "test classes within
a real show + rollback"
═══════════════════════════════════════════════════════════════════════════

Started by re-reading the Saratoga scenario. I proposed straight
mtime gating per docs/v3-planning/STALE-CLS-PREVENTION.md. Bill
pushed back: "the idea was that the website follows the classes
folder but that wont work in the case of the broken computer and
change." Genuine architecture concern — folder-as-truth breaks
when computers swap mid-week.

We talked through:

  • Two-way write — engine pulls files DOWN from website on a fresh PC
    rather than just pushing up. Folder becomes a working scratchpad;
    website is canonical. Deletions never auto-propagate.
  • Sunday-eager-delete scenario — Bill flagged the case where an
    operator clears Ryegate Sunday afternoon to set up Week 2 and
    inadvertently wipes the website. Resolved by rule: deletions are
    operator actions only (admin), never sync actions.

Then the test-flag pivot. I initially built shows.is_test (entire
sandbox shows like smoke-show). Bill corrected: "i was specifically
talking about having a CLASS named test within the actual show slug
that would only be shown in the test url." So we added classes.is_test
as the SECOND flavor and kept shows.is_test as a useful tool for
sandbox shows the operator wants to keep around.

Two-flavor model locked:
  • shows.is_test — entire sandbox show, hidden from public homepage,
    direct URL still works. Auto-detected from /test/i in show name.
  • classes.is_test — individual test class inside real show, hidden
    from public class list, revealed by ?test=1. Auto-detected from
    /test/i in class name on every postCls.

Date-gate rule locked:
  • No grace at start (Bill: "we need to gaurd against classes created
    before the date of the show no grace")
  • 24h grace at end (trailing finalize writes)
  • TEST show or TEST class → bypass entirely
  • TEST visible to operator only — never on public homepage

ENGINE 3.1.7 — date gate (the actual Saratoga fix)
  Bundled in commit bf17ed8.

  • showMeta cache (slug, start_date, end_date, is_test) fetched on
    startup, on show change, every 5 min. Sticky — transient fetch
    failure keeps the gate live with prior good values.
  • shouldUploadCls(filename, mtimeMs, bytes) returns { allow, reason }.
    TEST show / TEST class / date-ok / before-start-date / after-end-date+grace.
  • Gate runs in postClsFile before every upload. Blocked files counted,
    logged, pushed to renderer.
  • Class-name detection via tiny inline CSV-aware parser
    (parseCsvLineEngine + parseClsClassName) — engine peeks col[1] of
    row 0 without round-tripping the worker.

ENGINE 3.2.2 — rollback support
  Commit: a1fff49

  Three pieces:

  1. Preservation: installUpdate's swap batch now renames the current
     app.asar to app.asar.previous before installing the new one.
     Sidecar previous-version.txt records what's in .previous. Both
     live in c:\west-engine\resources\.

  2. Manual rollback: Settings → Updates → "Roll back to X.Y.Z" row
     under Check now. Disabled with hint until first OTA preserves
     a previous version. Click writes rollback-swap.bat that swaps
     asar ↔ asar.previous via temp name, exits, batch relaunches.
     Sidecar rewritten with the version we just left so the operator
     can swap forward again.

  3. Auto-rollback on crash-loop: handleFatal, when the 3-crashes-in-60s
     guard fires, checks previousVersionInfo() and calls rollbackEngine()
     one time before giving up. Clears crash log so rollback target gets
     fresh budget. Doesn't catch startup-class crashes (the handler is
     inside main.js, never registers if main.js can't load) — explicit
     limit documented in release notes.

  Smoke test added: `npx asar list <version>.asar` before every publish.
  Verified 3.2.2 includes reconciliation.js before deploying.

LESSONS / GOTCHAS
═══════════════════════════════════════════════════════════════════════════

1. ELECTRON-BUILDER FILES LIST IS AN ALLOWLIST
   New top-level engine modules MUST be added to v3/engine/electron-builder.yml
   files: section. Glob defaults don't catch them. Smoke-test via
   `npx asar list dist/release-asar/<version>.asar` before any publish.
   Saved as feedback_electron_builder_file_list.md.

2. UPDATE-WORKER-MANIFEST REGEX BREAKS ON APOSTROPHES
   v3/engine/build/update-worker-manifest.js uses
   /releaseNotes:\s*['"`]([^'"`]*)['"`]/ which cuts off at the first
   apostrophe inside the notes. 3.1.7 publish failed once when I wrote
   "show's" in the notes — the script preserved "show" then wrote
   "show\'," producing an unterminated string literal in west-worker.js.
   Workaround: don't use apostrophes in releaseNotes. Real fix is a
   smarter regex; deferred.

3. PUBLIC_PAGES_ORIGIN STILL POINTS TO PREVIEW
   v3 hasn't cut over to westscoring.live yet. The engine's test URL
   is currently preview.westscoring.pages.dev/show.html?slug=… — same
   domain the admin link opens. Constant at top of main.js to swap when
   v3 goes production.

4. CHECKBOX INPUTS STRETCH WITH .form-group input { width: 100% }
   Admin show edit "Test show (sandbox)" checkbox initially looked
   centered/floating because the global rule applied. Added
   .form-group input[type="checkbox"] { width: auto; padding: 0; }
   override.
