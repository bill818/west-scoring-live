WORKER (west-worker.js)

  ── /scores-update empty-overwrite guard (b95f9de) ───────────────────────
  Bill 2026-05-08: "class 326 should have standings." Snapshot had
  jumper_scores=[] despite D1 having data. Cause: pullJumperScoresV3
  sometimes returns [] mid-stale-sweep (from /v3/postCls), and that []
  was clobbering populated state via the existing /scores-update write.
  Guard: if incoming scores is empty AND prior is populated, keep prior.
  Self-healed within seconds of deploy.

  ── _updateByClass: pickScores() lens routing + empty-overwrite (fcb3fa2)
  Bill: "class 341 in saratoga the live just randomly lost their standings
  after 3 horses." Same shape as 326 but on a different write path. Body
  score arrays were being written into byClass[focused_class_id] regardless
  of which class the body's scores belonged to. When operator clicked
  another class on Ryegate, focused flipped, the next /v3/postCls write
  silently clobbered the wrong slot.

  Fix: new `pickScores(bodyScores, priorScores, bodyClassMeta, focusedClassId)`
  helper. Routes to byClass[bodyClassMeta.class_id] (the LENS of the
  incoming body), with the same empty-overwrite guard. Page-rendering
  decision (focused) decoupled from write-target decision (lens).

  Captured in feedback_lens_class_routing.md. The lens IS the body — focus
  is for what to broadcast back, never what to write.

  ── _decodeHunterDisplayedRound + previous_entry hunter fields (1eafefe)
  Bill 2026-05-08: "for overall that seemed like a patch not a fix... yes
  fix it right while we have test shows to work with."

  First attempt to render hunter just-finished was client-side: live.html
  was guessing whether the announced score was R1, R2, R3, or Overall by
  comparing combined_total to individual rounds. That bled classification
  logic into the page and got it wrong on edge cases.

  Refactored to worker-side helper `_decodeHunterDisplayedRound(row,
  classMeta)`:
    1. Single-round match: combined_total == one r{N}_h_score → that's it.
    2. Otherwise iterate non-empty subsets sized 2..N, find the LARGEST
       subset whose sum equals combined_total → Overall.

  Surfaced on previous_entry as `displayed_score`, `displayed_round_label`,
  `displayed_is_overall`. _buildPrevEntry signature now takes classMeta as
  third arg so it can call the decoder. Captured in
  feedback_hunter_displayed_round.md.

PRINCIPLES REINFORCED
═══════════════════════════════════════════════════════════════════════════

1. SAME TEMPLATE + SAME OPTIONS = SAME OUTPUT
   class.html and live.html both call WEST.jumperTemplates.renderTable.
   Today live.html had drifted to passing fewer options — and silently
   produced different output. Rule tightened in
   feedback_use_centralized_templates.md: pass the SAME options the
   reference page passes. Page-specific tweaks ride on the options
   object, never as parallel render paths.

2. THE LENS IS THE BODY
   Body score arrays belong to the class the LAST UDP event was for —
   never to whatever class the operator happens to be focused on. This
   was the root cause of two separate "standings disappeared mid-class"
   bugs (326, 341). pickScores() at the boundary; never re-introduce
   "fall back to focused" routing.

3. DECODE IN THE WORKER, NOT THE PAGE
   Hunter Overall vs single-round classification belongs in
   _decodeHunterDisplayedRound. Live page reads displayed_round_label
   directly. Same rule for any "what was actually announced?" decision —
   live.html, results.html, and the future ring-display page must all
   render the same answer, which means decoding is centralized.

4. EMPTY ARRAYS DON'T OVERWRITE POPULATED ARRAYS
   Across multiple write paths, an [] mid-stale-sweep silently clobbered
   real data. Guard at every boundary, not above. Both /scores-update
   and _updateByClass got the guard today.
