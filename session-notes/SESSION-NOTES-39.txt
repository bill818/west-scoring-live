PART A — CLEAN URLS ATTEMPTED, DEFERRED
═══════════════════════════════════════════════════════════════════════════

Bill: "lets try the clean urls".

Built `_redirects` with Pages 200-status rewrites + named placeholders
(`/v3/:slug/:ring/:class → /v3/pages/class.html`). Tested against
real URLs and discovered Cloudflare Pages doesn't actually do internal
rewrites for placeholder rules — it 308 redirects to the destination,
strips the `.html` extension via auto-canonicalization, then matches
the no-extension URL against the same `_redirects` rules → infinite
loop (`/v3/pages/class` → `/v3/pages/ring` → `/v3/pages/ring` → ...).

A pass-through identity rule (`/v3/pages/* → /v3/pages/:splat 200`)
broke the loop, BUT the address bar still showed the ugly destination
URL after the 308 — and the destination page had no slug/ring/class
info because the rewrite stripped the original path.

CONCLUSION: `_redirects` placeholder rewrites are advertised as URL-
preserving but operate as 308 redirects in practice. Address-bar-clean
URLs require Cloudflare Pages Functions (a tiny inline worker that
does a real internal rewrite + serves the static asset).

DOC: `docs/v3-planning/CLEAN-URLS-DEFERRED.md` — full context, what we
tried, why it failed, code sketch for the Functions-based fix that
ships pre-go-live.

KEPT: `WEST.api.urls.{index,admin,show,ring,cls}` URL builders +
`WEST.api.pathParam(name)` reader in `v3/js/west-api.js`. Harmless
no-ops on the current site, ready for the Functions revisit.

REVERTED: link-site changes (back to query strings), `_redirects` file
deleted, deploy script changes rolled back, getQueryParam ↔ pathParam
swap reverted on each page.

Trigger: public go-live work (after engine connect + v2 retirement).

PART B — JUMPER STATS DESIGN (STEP 1, PAPER)
═══════════════════════════════════════════════════════════════════════════

Doc: `docs/v3-planning/JUMPER-STATS-DESIGN.md` (~600 lines).

Architectural mirror of the hunter pre-compute pattern:

  HUNTER                                JUMPER ANALOG
  computeJudgeGridRanks                 computeJumperStats
  migration 018                         migration 020 (this session)
  /v3/postCls hook (H branch)           /v3/postCls hook (J/T branch)
  /v3/listJudgeGrid                     /v3/listJumperStats
  /v3/recomputeJudgeRanks               /v3/recomputeJumperStats

Lock-ins captured in the doc:

- ONE row per class in `class_jumper_stats`. Per-round R1/R2/R3
  numbered slots match how `entry_jumper_rounds` is structured.
  Consumer labels rounds via `WEST.format.roundLabel` (no labels
  in the table).
- Mode-aware fault histogram via JSON `r{N}_fault_buckets TEXT`
  columns. Schemes: `standard` / `speed` / `optimum` / `none`.
  Schema is static; bucket DATA inside the JSON varies per scheme.
- Backfill via `/v3/recomputeJumperStats` (same pattern as
  `/v3/recomputeJudgeRanks`).

Bill chose A + C from my menu of "more dynamic":
  A = method-aware bucket DEFINITIONS (each scheme has its own labels
      + boundaries)
  C = JSON-column STORAGE (adding a scheme later = code change only)

Resolved with Bill on schemes:
  - `OC` = Off Course → eliminated family. `IN ('EL','RF','DNF','OC')`.
  - "Time fault" = `r.time_faults > 0` (any). Counts entries that
    exceeded TA (jump faults can coexist).
  - Standard scheme expanded to 8 buckets with 1-3 and 5-7 splits to
    distinguish time-fault-only from clean-rail patterns.
  - Render rule: zero-count buckets always written (comparable across
    classes); frontend filters at render time so chip strip is
    clutter-free.

PART C — JUMPER STATS IMPLEMENTATION (STEPS 2-3)
═══════════════════════════════════════════════════════════════════════════

MIGRATION 020 — `class_jumper_stats`
  - One row per class, FK on class_id
  - Aggregate counts (total / scratched / eliminated)
  - Per-round R1/R2/R3 columns: competed, clears, time_faults,
    avg_total_time, avg_clear_time, fastest_clear_entry_id,
    fastest_clear_time, fault_buckets (TEXT JSON)
  - computed_at metadata

MIGRATION 021 — fastest_clear → fastest_4fault rename
  Bill: "show fastest 4 faulter instead of fastest clear"
  Six column renames via SQLite ALTER TABLE RENAME COLUMN.
  Filter changes per scheme:
    - standard: total_faults = 4 (one rail clean)
    - speed:    jump_faults = 4 (one rail; time penalty in total_time)
    - optimum / none: NULL (concept doesn't apply)

MIGRATION 022 — entry-list stats (jumper-only)
  Bill: "lets call these entry specific stats cause they will populate
  first before any data even starts flowing in" → "no dont apply these
  to hunters this will stay in the jumper side only"
  Five new columns on `class_jumper_stats`:
    unique_riders / unique_horses / unique_owners
    countries_json    — [{code, count}] desc by count
    multi_riders_json — [{rider, horses:[...]}] desc by count

WORKER — `computeJumperStats(env, classDbId)`
  - Lives next to computeJudgeGridRanks
  - Mode-aware: `bucketSchemeFor(method, modifier, round)` picks per
    round. Method 6 modifier=1 mixes optimum (R1) + standard (R2 = JO).
  - Helpers: `computeJumperRoundStats`, `computeJumperBuckets`,
    `computeJumperEntryStats`, `parseStatsJson`
  - Hooked into `/v3/postCls` jumper branch: synchronous after the
    UPSERTs; failures log + entriesStatus, never fail the POST
  - DELETE-then-UPSERT pattern (full row refresh; no stale columns)

WORKER — `/v3/listJumperStats?class_id=N`
  Single read endpoint. Returns one envelope:
    { ok, class:{...}, stats:{ counts, rounds:[{round, ..., standings:[]}],
                              entry_stats:{ unique_*, countries, multi_riders } } }

  Standings — fully server-rendered. Two SQL paths picked per round:

  Default mode (gap from leader):
    - Place: RANK() over (total_faults asc, total_time asc) for clean
    - Killed entries excluded from RANK pool via PARTITION
    - Gap class/display: leader / behind (+X.XXXs amber) /
      fault-gap (+N flt red) / "—" for killed
    - Status filter: NOT IN ('DNS','WD','SC') (didn't compete)
    - Killed entries (EL/RF/DNF/OC) listed below clean rides

  R1 of multi-round methods (II.2 family + 3R + Two-Phase + Winning
  Round + Optimum 2R) — JO mode:
    - Bill: "for these II.2 family (and fei equivalent) Round 1 should
      have distance from TA, not leader and gap"
    - Place: 'JO' chip for entries at min_faults; ROW_NUMBER over
      non-killed entries gives stable position numbers (no RANK gaps
      from killed entries) — first 4-faulter is rank 18, not 19+
    - Sort within JO qualifiers: ride_order ASC (Bill: "perfer to keep
      the R1 Clears in the order they went in if possible")
    - Sort within faulted-non-killed: faults asc, time asc
    - Gap = signed distance from r1_time_allowed:
      "Gap from TA - 73s" header label per round (server-computed)
      gap_class: leader (>3s under, green) / behind (0-3s under, amber)
                 / fault-gap (over TA, red) / "—" for killed

  Every display string SQL-rendered: place_display, place_class,
  fault_class, fault_display, time_display, gap_class, gap_display,
  gap_label. Bill: "id like it to not be done client side, harder
  build now easier to cherry pick stat boxes when we build out other
  pages."

WORKER — ETag support
  `/v3/listJumperStats` uses the existing `jsonWithEtag(request, data)`
  helper. CORS expanded with `If-None-Match` (Allow-Headers) + `ETag`
  (Expose-Headers) so browser polls don't 403.

WORKER — `/v3/recomputeJumperStats` safety valve
  Body shapes:
    {}                  → recompute every J/T class
    { slug: 'X' }       → scoped to one show
    { class_id: N }     → single class
  Idempotent. Used post-deploy to backfill the 51 existing T classes.

41. CLEAN URLS NEED PAGES FUNCTIONS, NOT _redirects.
    Cloudflare Pages 200-status `_redirects` with placeholders
    operates as a 308 redirect in practice (auto `.html` strip causes
    re-walk + loop without a pass-through; pass-through fixes loop but
    address bar still shows destination). Pages Functions is the only
    way to get true URL-preserving rewrites. Documented in
    `docs/v3-planning/CLEAN-URLS-DEFERRED.md` for go-live work.

42. JUMPER STATS PRE-COMPUTED, NOT CLIENT-COMPUTED (mirrors session-36
    rule for hunter). class_jumper_stats holds counts, per-round
    aggregations, server-rendered display strings. Pages render
    verbatim — every value comes pre-formatted (place_display,
    fault_class, fault_display, time_display, gap_class, gap_display,
    gap_label). Cherry-pickable: future pages drop in one row's HTML
    without re-implementing display logic.

43. ENTRY-LIST STATS ARE JUMPER-ONLY.
    Bill 2026-04-26: "no dont apply these to hunters this will stay
    in the jumper side only." unique_riders / horses / owners +
    country breakdown + multi-ride riders all live on
    class_jumper_stats. Hunter analog (when wanted) would land on a
    parallel hunter table.

44. R1 OF MULTI-ROUND METHODS USES JO MODE STANDINGS.
    Bill: "Round 1 should have distance from TA, not leader and gap."
    R1 of methods 2/3/9/10/11/13/14/15 + 6+modifier=1 swaps gap-from-
    leader for distance-from-TA, JO badge for entries at min_faults,
    qualifiers sorted by ride_order. R2/R3 (placing rounds) stay on
    default gap-from-leader mode.

45. ETag-AWARE POLLING IS THE SMART-POLL PATTERN.
    Bill: "could there be something smarter? the worker... knows
    everything". Server sets ETag from the response body hash; client
    polls every 10s with If-None-Match; worker returns 304 when nothing
    changed. Most polls are tiny round-trips (~hundred bytes each).
    Real push (WebSocket) waits for Durable Objects in Phase 7-8;
    endpoint contract stays stable so the swap is invisible to
    consumers.

46. west-display.js IS THE HOME FOR FUTURE CHIP PATTERNS.
    Today's helpers (chip, statusBadge, placeMarker, countryChip) are
    stable. Add new helpers when a new chip pattern appears across
    pages. Don't retroactively refactor settled chip CSS just to use
    the helper — visual stability wins over uniformity.

47. JUMPER-ONLY STATS BUTTON ON CLASS.HTML HERO.
    `📊 Class Stats` button rendered in renderHero only when
    cls.class_type ∈ {J, T}. Hunter classes never see the button
    because the endpoint is jumper-only.
