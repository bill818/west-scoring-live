HEADLINE
═══════════════════════════════════════════════════════════════════════════

Phase 2d (jumper half) shipped end-to-end in one session. What used to
be v2's "parse the .cls and guess at per-round statuses" is now a
structurally clean pipeline:

  engine .cls POST
     → worker parseClsHeaderV3 (picks lens from col[0])
     → worker parseClsEntriesV3 (identity cols 0-12)
     → worker parseEntriesScoreJ (J or T lens) — per-round independent
     → UPSERT entry_jumper_scores keyed by entry_id
     → /v3/listEntries returns joined scoring to admin
     → admin renderEntries applies Decision 1 (cell suppression) +
       Decision 2 (place suppression on R1 kill)

959 rows across 51 classes backfilled via a single engine startup-sync
with zero parse warnings and zero failures. Verified against a Ryegate
published PDF for class 23 — every placement, time, fault count, and
status matched (one benign "richer" difference: v3 stores RF where
Ryegate's PDF shows the generic "EL" — expected behavior).

The foundational v2 parser bug is fixed: v2's "attribute single overall
status to all rounds" logic produced double-WD artifacts (e.g., entry
1993 EXCELLENT in Culpeper class 212 had r1_status="WD" stored even
though R1 was clean). v3's per-round independent attribution stores
r1_status=NULL, r2_status="WD" — honest to the raw .cls.

THE V2 PARSER BUG — DIAGNOSED AND FIXED STRUCTURALLY
═══════════════════════════════════════════════════════════════════════════

Investigation surfaced a real bug in the v2 production data:

  v2 production DB stores for Culpeper class 212 entry 1993:
    r1_status = 'WD'    ← WRONG
    r2_status = 'WD'    ← right

  Raw .cls for that entry (pulled from classes.cls_raw in v2 DB):
    col[21] = 0         (no R1 status)
    col[28] = 4         (R2 numeric status = WD)
    col[37-39] all empty (no text status anywhere)

  Ryegate wrote ONE signal: col[28]=4. R1 was genuinely clean
  (time 73.626, 0 faults). v2 watcher's line 1005 computes a single
  overall entry.statusCode; downstream storage mechanism writes that
  single value to BOTH round rows in the results table.

Same pattern confirmed across:
  - Class 212 entries 1993, 6191, 6267 (triple-WD after clean R1)
  - Class 264 entry 6056 (OC stored on both R1 and R2)

v3's structural fix:
  - Parser never computes an overall statusCode — each round parsed
    independently from its own numeric + the (attributable) tail text.
  - Storage target is the linked entry_jumper_scores table, one row
    per entry — but with per-round columns (r1_status, r2_status,
    r3_status) that get written independently from the parse output.
  - No single "statusCode" field in the DB at all. If any future
    consumer wants "overall status," it derives it at render time
    as r2_status || r1_status || r3_status || null.

Public-facing behavior unaffected (v2's display worked around the bad
storage by checking time-presence). v3 simply stores truth.

ARCHITECTURAL DECISIONS LOCKED THIS SESSION
═══════════════════════════════════════════════════════════════════════════

1. LINKED LENS TABLES, NOT WIDE SINGLE-TABLE.
   Rationale: stats correctness. Wide-table AVG() silently drops NULLs
   which would produce wrong cross-lens averages. Linked tables force
   queries to join through a specific lens table, making wrong-lens
   queries impossible to write by accident. Per-lens column evolution
   also stays independent. Bill's instinct on this was correct;
   Claude had to be convinced and reconverted.

2. LENS MODULE IS VERSIONED.
   west-cls-jumper.js stores every layout version that's ever
   existed. Old archives re-parse against the layout they were
   written with. When Ryegate shifts columns, add a new version,
   don't mutate old ones.

3. PER-ROUND STATUS IS INDEPENDENT — ALWAYS.
   No overall statusCode anywhere. Each round's text + numeric
   attributed from its own columns (or from the tail-scan attributed
   to the LATEST fired numeric for Farmtek). Never propagated.

4. HASGONE (col[36]) IS UNRELIABLE, NOT STORED.
   Cross-confirmed across 2,357 rows (Farmtek + TOD) that col[36]=1
   can appear with zero scoring evidence (7 such entries found). Not
   a "show on results" flag either — three Farmtek anomalies verified
   on Ryegate's live results page, none displayed. Parser ignores
   col[36]. "Did they ride" is derived from (time > 0 OR status) at
   render time.

5. NUMERIC STATUS MAP IS EMPIRICAL, NOT SPECULATIVE.
   JUMPER_NUMERIC_MAP starts with {2:'RT', 3:'EL', 4:'WD'} —
   ONLY codes observed live in Culpeper data. Unknown values
   (1, 5, 6) log parse_warning on first sight. Map expands from
   evidence. Session 22's speculative "3=HF" and display-config.js's
   speculative "3=OC" were not used.

6. SPECIFIC CAUSE CAPTURED INTERNALLY; PUBLIC DISPLAY GENERIC.
   Per Bill's directive: admin/internal shows specific codes
   (RF/HF/OC/RO/DQ); public pages collapse to generic "EL". Not a
   per-code publicLabel field — one-line render helper at the
   public page level when Phase 2e lands:
     publicLabel(code) = TEXT_CODES[code].category === 'ELIM' ? 'EL' : code

7. DISPLAY LAYER DECIDES PLACEMENT, NOT PARSE.
   overall_place is stored raw from col[14] exactly as Ryegate
   stamped it. Admin's Decision 2 suppresses place when R1 has a
   killing status (approximation of the full method-aware ladder
   logic). Proper effectivePlacement() function with method-table
   walk and wipesOnFail ladder rules lands in west-rules.js in
   Phase 2e.

8. DECISION 1 SUPPRESSES ROUND DATA WHEN STATUS FIRES.
   Round cell shows either time/faults OR the status label —
   never both. Matches Ryegate PDF convention. Raw time/faults
   stay in the DB for future use (partial-ride stats etc.).
