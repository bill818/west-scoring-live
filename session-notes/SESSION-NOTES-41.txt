7. PER-SHOW ENGINE LOCK — full pattern (the big one)

   Schema (migration 027):
     shows.lock_override TEXT NOT NULL DEFAULT 'auto'
       CHECK IN ('auto', 'unlocked', 'locked')

   Effective lock:
     'locked'   → always locked
     'unlocked' → never locked
     'auto'     → locked iff end_date < today (UTC string compare)

   Worker plumbing (west-worker.js):
     - computeShowLock(showRow) → { locked, reason }
       reason is one of: 'manual' (lock_override='locked'),
                         'auto-after-end-date',
                         null (unlocked).
     - isShowLockedV3(env, slug) → boolean (convenience).
     - lockedResponse(reason) → 423 Response with
       { ok:false, locked:true, reason, error: 'Show is locked from engine writes' }

   Enforced on engine-write endpoints:
     POST /v3/postCls
     POST /v3/deleteCls
     POST /v3/postTsked
     POST /v3/engineHeartbeat

   Lock check rolls into the same SELECT that validates show/ring
   (one round-trip — no extra query). Pattern:

     SELECT r.id AS ring_id, s.id AS show_id, s.lock_override, s.end_date
     FROM rings r JOIN shows s ON s.id = r.show_id
     WHERE s.slug = ? AND r.ring_num = ?

   Admin endpoints bypass the lock by design — the lock is about
   engine writes, not operator intervention. Endpoints that bypass:
     - hardDeleteCls (operator deletes a soft-deleted class)
     - recomputeJudgeRanks / recomputeJumperStats / reparseClassHeaders
     - updateShow / updateRing / createShow / createRing
     - downloadCls (read-only)

   API surface:
     - listShows now emits is_locked (0/1) AND lock_override (raw).
     - getShow emits is_locked.
     - updateShow accepts lock_override and validates the enum.

   Admin UI:
     - Show edit dialog (dlgShow) has a 3-way "Engine lock" dropdown
       at the bottom: Auto / Unlocked / Locked, default Auto.
       Hint text mentions HTTP 423 and the "morning-after correction"
       use case for Unlocked.
     - Sidebar shows 🔒 next to locked shows. Tooltip names the
       lock reason ('manual' vs 'auto-after-end-date').

   Operator escape hatch: flip to "Unlocked" to make corrections
   without rolling back end_date.

ARCHITECTURAL RULES LOCKED THIS SESSION
═══════════════════════════════════════════════════════════════════════════

1. SERVER IS AUTHORITATIVE FOR RANKS. When porting v2 logic that
   resorted by raw totals on the client, prefer reading the
   pre-computed rank columns. By-judge view sorts by judge_card_rank
   ASC, not judge_card_total DESC. Tied ranks become a small known
   ambiguity in client display — fine for now.

2. ONE LOCK-CHECK PATTERN ACROSS ENGINE ENDPOINTS. Pull
   lock_override + end_date in the same SELECT that validates
   show/ring. Call computeShowLock(row). If locked, return
   lockedResponse(reason). Template applies to any new engine
   endpoint added later.

3. PAST GROUPING IN ADMIN IS CALENDAR-AWARE. Status field is the
   authoritative manual state; sidebar grouping is a presentation-
   layer enrichment that respects end_date. Public pages have always
   been calendar-driven; admin is now aligned at the bucket level
   even though the underlying status field is still operator-owned.
