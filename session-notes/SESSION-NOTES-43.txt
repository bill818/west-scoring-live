The full UDP-to-spectator push pipe is alive end-to-end in a single session.
8 chunks shipped across two phases. Bill's "i cant wait a year" timeline
(per project_websocket_rebuild.md) compressed into a single afternoon
because the engine groundwork from S42 was already solid.

Subjective verification: Bill's report on the live.html visual after
WebSocket landed — "so much smoother." The polling jitter that motivated
3b is gone. Underlying 1Hz data cadence still applies; if sub-second
visual smoothness becomes a requirement, port v1.11's tickOnCourse /
applyHeartbeatClock interpolation pattern into a new west-clock.js v3
module. (Not done this session — flagged as next polish chunk.)

End-to-end pipe today:

  Ryegate UDP (1Hz) →
  Engine batcher (250ms windows) →
  POST /v3/postUdpEvent →
  RingStateDO.fetch('/event') in-memory + KV mirror →
  RingStateDO broadcasts to every connected WebSocket →
  live.html receives + renders (~50-100ms after engine sees the frame)

Polling stays as a fallback transport inside live.html (S30 cost-risk note:
corporate wifi / mobile carriers sometimes block WS).

ARCHITECTURAL RULES LOCKED THIS SESSION
═══════════════════════════════════════════════════════════════════════════

1. ENGINE CARRIES NO LENS
   makeUdpEvent does NOT set lens (jumper / hunter / equitation). Engine
   knows class_id; worker resolves lens by joining classes.class_type
   (and eventually class_kind). Per Article 1, lens is determined by the
   .cls's classType, not by UDP shape.

2. SLUG + RING ARE BATCH-LEVEL, NOT EVENT-LEVEL
   Each batch is guaranteed slug/ring-uniform because stopUdpListeners
   drops the in-flight batch on show-switch. No need to repeat slug/ring
   on every event. Reduces wire payload, simplifies validation.

3. NO RETRY ON DROPPED BATCHES
   A 250ms-window batch represents live truth at that moment. Replaying
   stale batches confuses snapshot semantics — next batch arrives in
   250ms anyway. Engine drops batches when liveScoringPaused, showLocked,
   or !configReady. Heartbeat carries lock-release detection.

4. DO IS THE SINGLE WRITER
   /v3/postUdpEvent routes through env.RING_STATE — DO mutates in-memory
   snapshot AND writes KV mirror. No two-writer race. KV stays the
   read-fallback surface (polling endpoint + Chunk 8 polling fallback).

5. HIBERNATION IS NON-NEGOTIABLE FOR PRODUCTION COST
   Without hibernation API, DOs are billed for duration during every
   open WebSocket. 500 spectators × 8hr × 5 rings would dominate cost.
   With hibernation, DO is only billed for actual CPU time (when a
   message processes). Use state.acceptWebSocket(ws), not ws.accept().
   Use webSocketMessage / webSocketClose / webSocketError handler
   methods, not addEventListener. Connections survive DO eviction.

6. RENDER() IS THE TRANSPORT-AGNOSTIC CONTRACT
   Both WS messages and poll responses call render(snapshot) with
   identical shapes. Future transport changes (SSE, push notifications,
   anything) just need to feed render(). No UI changes ever.

7. POLLING FALLBACK IS NOT WASTE
   Some networks (corporate wifi, some mobile carriers) block WebSocket.
   Polling stays as the live.html fallback for those clients — same
   /v3/getRingState endpoint, no extra worker code. Polling is also how
   we cold-warm the page if the WS handshake takes >2s.

KNOWN BUGS / EDGE CASES
═══════════════════════════════════════════════════════════════════════════

1. SUB-SECOND VISUAL SMOOTHNESS NOT YET DONE
   Ryegate sends frame=1 at 1Hz. Even with WS push, displayed clock
   values change 1×/sec. v1.11's interpolation (tickOnCourse) is the
   pattern to port — port into a new v3/js/west-clock.js that takes
   the latest snapshot and ticks at ~30Hz between data points. Not done
   today — Bill confirmed visual is "so much smoother" without it, but
   if a real show wants smoothness on par with v2 live.html, this is
   the chunk.

2. CLS POSTS LOOK 1:1 WITH UDP TICKS DURING STARTUP
   Misleading correlation — startup re-sync of all .cls files runs at
   150ms pacing, happens to interleave with UDP batches, but they're
   independent. Steady state has 0 .cls posts unless Ryegate actually
   modifies a file. Documented as Bill's question this session.

3. D1 udp_events RETENTION NOT YET FILTERED
   Stores every event from every batch. Could grow fast. Phase 3b
   was meant to add events-only filtering at the DO layer; we didn't
   land it because volume during this session was bounded and the
   filter design needed real data observation. Add when D1 row count
   becomes a concern, OR when adding the observability dashboard.

4. WS DOESN'T HOLD LATEST FOCUS BEFORE FIRST CHANNEL B EVENT
   If a client connects before the engine has emitted any Channel B
   event, last.class_id might lag the actual operator focus. Becomes
   non-issue once any focus packet has been seen for the ring.

5. ws-test.html IS A VERIFICATION ARTIFACT, NOT PRODUCTION
   Useful for sanity-checking WS pipe health, but it's intentionally
   ugly + not linked from anywhere. Could move to v3/tools/ or delete
   in a future cleanup. Left for now because it's the simplest "is the
   WS pipe working?" tool.

6. BATCHER CLEARS ON STOP — events between switch-show and re-bind
   stopUdpListeners drops the in-flight batch (correct: events tagged
   with stale slug/ring shouldn't ship). But this is the only
   guaranteed-clean state — if the operator switches shows mid-frame
   AND the new ring is on a different Ryegate cycle, the first batch
   from the new ring may be slightly delayed. Not a bug, just timing.

WHAT NOT TO DO
═══════════════════════════════════════════════════════════════════════════

- Do not put slug/ring on individual UDP events. They ride at the
  batch level. stopUdpListeners guarantees uniformity.
- Do not log successful UDP batch POSTs. ~14k/hr at 250ms cadence —
  use the state counters, not engine_log lines.
- Do not retry dropped batches. Each batch is a live-truth slice;
  replaying old events confuses the snapshot semantics.
- Do not bypass env.RING_STATE in /v3/postUdpEvent. The DO is the
  single writer for the KV mirror. Two writers = race conditions.
- Do not use ws.accept() in the DO. Use this.state.acceptWebSocket(ws)
  for hibernation. Production cost depends on it.
- Do not put the lens determination in the engine. Engine knows
  class_id; lens lives on the worker side via the classes table.
  Article 1 applies to UDP routing too.
- Do not assume tag X means the same thing across frames. Article 1
  generalizes to UDP — see S42 rule 6.

ARCHITECTURAL RULES LOCKED THIS ADDENDUM

1. CLOCK MODULE IS GENERIC; LENS LIVES IN THE PAGE
   west-clock.js doesn't know about jumper vs hunter vs equitation.
   It just owns Channel A frame 1 + frame 0 logic. The page reads
   class_kind from the snapshot and overrides the clock display when
   the lens has no clock semantics (hunter today; future: maybe
   "judged" lens for ribbons-only classes). Keeps the module reusable.

2. SAME-VALUE NO-REBASELINE
   When a server update lands with the same baseValue we already had,
   keep the existing baseAt. Otherwise visual interpolation restarts
   every second of stuck data. Phase tracking still updates so we
   don't miss boundary transitions.

3. SNAPSHOT.LAST IS NOT ENOUGH
   Real-world batches mix Channel A frames and Channel B focus events.
   `last` (whichever came last in the batch) doesn't capture the
   meaningful state. Use last_scoring (Channel A) and last_focus
   (Channel B), and have the DO carry them forward across batches.

4. FRAMES 12 + 16 ARE TRIGGERS, RENDER ACCORDINGLY
   The page MUST surface a chip / placeholder for these frames, never
   raw tag values. The numeric content cycles across multiple display
   pages and the .cls is the authoritative source. Honest "no usable
   data here" beats misleading rotating values.

5. CLASS_KIND IS WORKER-DERIVED (NOT ENGINE-INFERRED)
   /v3/postUdpEvent enriches the snapshot with class_kind by joining
   classes via the focus class_id. Engine never tries to determine
   lens. Per Article 1 + S43 Chunk 1 commitment.

6. RUNNING-TENTH DEFAULTS TO ON
   Bill's preference 2026-05-02. Operator opts OUT to whole-seconds-only
   on the live page via the engine's "Live page" pane. Worker also
   defaults true when the field is missing on the wire (legacy engines
   still get the better default).
