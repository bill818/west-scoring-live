DECISIONS MADE THIS SESSION (LOCKED)
═══════════════════════════════════════════════════════════════════════════

1. SEPARATE v3 DATABASE (west-scoring-v3), not same-DB-with-v3-flag.
   Reason: v2 has essentially no production data (the "one real show"
   turned out to be a test show, not a contracted event). The stats-
   across-eras argument for a shared DB evaporated. With separate DB:
   v3 gets optimal schema from day 1, zero risk to v2, cleaner retirement
   when v2 walls off.

2. v2 walls off at cutover (Bill, session 30). Code deleted or commented
   out entirely rather than kept in parallel forever. "We're good at the
   end."

3. v3 DB strategy extends: same as before (new tables only, never mutate
   v2 tables), but now in a separate DB. Historical data from v2 stays
   readable for parser validation / stats training (never written to).

4. v3 admin layout: master-detail (sidebar + main pane), not flat tabs.
   Reason: classes belong to rings, rings belong to shows; most admin
   interactions happen inside ONE show. Engine is ring-attribute, not a
   top-level entity. Tabbed sidebar order is Upcoming → Active → Past.

5. v3 admin + pages will have SAME visual language as v2. Same fonts,
   colors, logo. v3 is "rebuild under the hood, not redesign on surface"
   per locked v3 principle.

6. Worker auth continues unchanged (X-West-Key header, same key as v2).
   HTML page-level password lock is OFF for v3 admin for now; future
   phase may add a proper admin login. The worker endpoint auth and the
   page login are separate concerns even though they use the same key
   today.

7. V3_ENABLED feature flag is ON in production wrangler.toml. Reason:
   v3 endpoints (all /v3/*) don't collide with v2 URLs, so turning them
   on doesn't affect v2 behavior. All v3 paths return 404 if the flag
   were flipped off. When we retire v2, this flag becomes redundant and
   can be removed.

8. Engine version: "3.0.0-dev" during development. Flip to "3.0.0" at
   first real operator install. Bump from there on every deploy.

9. Polling cadence for admin: 10s for rings (aligns with engine 10s
   heartbeat), 30s for show list. WebSocket is Phase 6+ and doesn't
   change any of the consumer code because everything goes through
   loadShows() / loadRings() functions — swap transport inside those
   later.

10. "Results page = pure view of .cls" principle (session 29 Addendum C
    of V3-BUILD-PLAN) stays the north star for Phase 2+.

═══════════════════════════════════════════════════════════════════════════
GOTCHAS / LESSONS FROM THIS SESSION
═══════════════════════════════════════════════════════════════════════════

1. Wrangler OAuth tokens expire. Bill's token died mid-session. Fix:
   `wrangler logout; wrangler login` in the user's own terminal (needs
   browser). Claude Code's bash session can't do this alone.

2. PowerShell execution policy blocks unsigned wrangler.ps1 on default
   Windows. Two fixes: (a) use cmd.exe instead, or (b)
   `Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned`
   once per user. Either works.

3. The "one real show in v2 DB" was actually a test show Bill ran — not
   contracted data. That re-opened the DB-separation question and led to
   west-scoring-v3 as a separate database.

4. The session 29 lesson about ELECTRON_RUN_AS_NODE env var still applies:
   always unset it when launching Electron apps from Claude Code's bash.
   Didn't bite us directly this session (I defaulted to `env -u
   ELECTRON_RUN_AS_NODE`) but it's a permanent trap.

5. The initial tray-icon placeholder was blank (same bug as the spike —
   I embedded a broken base64 PNG). Fix: use assets/favicon.png copied
   into the engine folder and loaded via nativeImage.createFromPath().
   Apply the same pattern to any future Electron tray icons.

6. CORS is already permissive on the worker (`Access-Control-Allow-Origin:
   *`), which lets local file:// pages call the API directly. Good for
   development. Will need tightening before real production use — that's
   future work.

NEXT SESSION — PHASE 2 OPTIONS
═══════════════════════════════════════════════════════════════════════════

Three natural directions, any can go first:

OPTION A — Engine starts watching .cls files
  - Engine watches C:\Ryegate\Jumper\Classes\*.cls
  - On change, POST raw .cls bytes to a new /v3/postCls endpoint
  - Worker stores raw in R2 (first R2 work in v3) + parses + stores
    structured rows in new D1 tables (classes, entries, ...)
  - Results page becomes viable as a PURE VIEW of the parsed structured
    data (Addendum C principle in action)
  - RECOMMENDED FIRST — it produces a user-visible result earliest

OPTION B — Engine reintegrates UDP fan-out from the spike
  - Merge spike's synchronous UDP fan-out pattern into the engine's main
    loop alongside the heartbeat
  - Forward UDP to RSServer (keeps the scoreboard working)
  - THEN add the UDP parse-and-POST path once fan-out is confirmed
  - Worth doing before .cls if there's a real show on the horizon that
    needs the scoreboard working

OPTION C — Observability tables + parse_warnings infrastructure
  - Create the DATABASE-SCHEMA-EXPANSION Part G tables in WEST_DB_V3
  - Wire the worker to write parse_warnings on unknown quirks
  - Less user-visible progress but high leverage — once the parser starts
    running in Options A or B, the observability captures quirks
    automatically

Bill picks based on what feels productive, what has external pressure
(a show coming up = Option B), what feels risky-soonest.
