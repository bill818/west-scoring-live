DESIGN ARC
═══════════════════════════════════════════════════════════════════════════

Started by reading v2/display.html (~1080 LOC, inline templates for every
jumper/hunter/derby/equitation variant) and the v3 module stack we already
had. Bill's directive was clear early: "I'd tend to say build this right
from our original architecture — the live page is working really well, I
feel like there are centralized functions we already built that can
stabilize that page."

Initial build dropped a lot of v2's page-side logic in favor of:
  • snapshot.focus_preview for the on-course identity / clock / faults
    (worker decodes UDP tags, page just reads fields)
  • WEST.jumperTemplates.renderTable / WEST.hunterTemplates.renderTable
    for standings — same renderers class.html + live.html use
  • previous_entry.banner_slots iteration only (no page-side fallback)
  • New WEST.scoreboard.renderOOG (alongside existing renderJogOrder /
    renderStandbyList) for the kiosk OOG / Seen panel

The page took several refinement passes once it was live against
osf-spring-ii ring 1 during an active class:

  1. RANK double-prefix — fp.rank arrives as "RANK 5" sometimes; my
     code added "RANK " in front, displaying "RANK RANK 5". Strip the
     leading "RANK\s*" before rendering.
  2. Friendly show name in QR slot — fetched /v3/getShow once on boot,
     fell back to slug while in flight.
  3. TA chips capped to actually-used rounds — Ryegate writes r3_time_
     allowed on .cls headers even for 2-round methods. Source of truth
     is jumperTemplates.detect (1R/EQ → 1, 2R → 2, 3R/TEAM → 3) for
     jumpers, num_rounds for hunters. Same fix later mirrored on live.
  4. OOG: drop the upcoming list entirely. Bill's call after seeing
     the .cls's ride_order isn't really operator-curated and shouldn't
     mislead spectators. WEST.scoreboard.renderOOG now takes a
     hideUpcoming opt; display.html passes true, other consumers don't.
  5. Seen list in ring-entry chronological order. Required new D1
     column finished_at on entry_jumper_rounds + entry_hunter_rounds
     (migration 037), worker preserving the value across its DELETE-
     then-INSERT round-write pattern, exposing r*_finished_at via
     listEntries, and sorting by latest-finished_at descending so the
     most-recent rider sits at the top.
  6. Dark theme contrast pass — instrumented an in-page audit walking
     every text node, computing fg vs effective-bg with rgba alpha
     blending, flagging contrast < 3.0. Found .results-table
     .round-label-stack rendering near-black on dark (was --navy),
     .place-num same (--black), --text-faint too dim, --red too dim
     as text. Targeted overrides on the standings-scoped rules.
  7. Per-region rendering split. Initial implementation re-wrote the
     entire #app innerHTML on every render, causing the topbar to
     flicker when scores arrived. Split into four <div id="region…">
     wrappers, each with its own innerHTML hash guard. Wrappers use
     display:contents so the inner .topbar / .oog / .standings /
     .sidebar remain the grid items.
  8. Sticky standings header. Plain position:sticky on <th> wasn't
     pinning. Tracked through three issues: border-collapse:collapse
     (inherited from west.css — switched to separate on the standings
     table), overflow:hidden on the table (was there for border-radius
     clipping — switched to visible), and position:relative on thead
     (kept). All three needed to align before sticky engaged.
  9. Autoscroll regression after the region split. wireAutoScroll ran
     on every render and reset scrollTop=0; fixed by gating on
     classDbId change. Then updateRegion's innerHTML write was
     destroying #standingsScroll and resetting it; fixed by
     capturing+restoring scrollTop on rebuild.
 10. Clock flicker debug. With the page running stable but the clock
     bouncing between phases, instrumented the WS stream to capture
     last_scoring frames over time. Caught the clock's phase flipping
     between on_course/integer and finished/decimal every ~5s. Root
     cause was the keepalive poll I had added earlier as a zombie-WS
     safety net — it was fetching /v3/getRingState which returned a
     slightly-stale snapshot, re-baselining the clock module to a
     previous rider's decimal t17. Suppressed all polls at runtime
     and confirmed the flicker stopped. Replaced keepalive with a
     proper zombie-WS detector: if WS reports OPEN but no message
     arrives in 30s, force close so the existing onclose reconnect
     fires.
 11. Phase label stuck on "Phase 2". Two-Phase methods (method 9) ride
     both phases per rider — TA flips twice per ride. WEST.clock state
     roundNum = taSeen.length, which settles at 2 once both TAs are
     observed and stays there forever. Switched the page's round-tag
     derivation to taSeen.indexOf(currentTa) + 1 — "which TA is firing
     RIGHT NOW", not cumulative count.
 12. Bill noticed only 50 entries showing in the standings on a
     72-entry class. Asked "how many in the cls? is it fifty or more
     than fifty?" — pointed straight at the bug. Worker had hardcoded
     LIMIT 50 in both pullJumperScoresV3 and pullHunterScoresV3.
     Removed. Pill went from "48 of 50" to "48 of 72" on next refresh.

Bill's higher-order theme throughout: "are we using centralized and
tested calls for these pages? no client side formulations?" — pushed
hard on the discipline of building this page as a thin consumer of
shared modules + worker contracts, not as another bespoke surface.
The OOG renderer move into WEST.scoreboard, the focus_preview-only
on-course card, and the banner_slots-only previous card all came
out of that pressure.

LESSONS / GOTCHAS
═══════════════════════════════════════════════════════════════════════════

1. STICKY <th> NEEDS THREE THINGS TO ALIGN
   - border-collapse: separate on the <table>
   - overflow: visible on the <table> (not hidden — west.css had it
     hidden for border-radius clipping)
   - position: sticky + top: 0 on the <th>
   Any one missing kills the pin. Diagnosed by walking the ancestor
   chain looking for overflow / position contexts.

2. ROUNDNUM IS CUMULATIVE, NOT CURRENT
   WEST.clock._state.roundNum = taSeen.length. Settles at the highest
   round count observed for the class and stays there. For "what round
   is on the scoreboard right now", use:
     taSeen.indexOf(currentTa) + 1
   Critical for two-phase methods that flip TA per-rider.

3. KEEPALIVE POLLS POISON CLOCK STATE
   Adding a 5s keepalive fetch of /v3/getRingState as a zombie-WS
   safety net seemed harmless. It wasn't — the cached snapshot the
   endpoint returned could carry a STALE last_scoring (from a
   previous rider with a decimal t17), which when fed to
   WEST.clock.set re-baselined the phase to "finished" and overwrote
   the live on-course state. Every 5 seconds the clock would flicker
   to a previous rider's finish time and back. Zombie-WS detection
   should NOT replay snapshots; just monitor last-message timestamp
   and force-close stale sockets.

4. WORKER LIMIT 50 ON SCORES QUERY
   pullJumperScoresV3 + pullHunterScoresV3 had a hardcoded LIMIT 50
   that capped the snapshot's jumper_scores / hunter_scores arrays.
   Pill counts, standings table, ALL downstream consumers showed
   "X of 50" even for 72-entry classes. Real-world class sizes
   regularly exceed 50; the limit was a development-era artifact.

5. FULL-PAGE innerHTML REPLACE ON EVERY WS MESSAGE = FLICKER
   Initial display.html did a single document.getElementById('app').
   innerHTML = html on every snapshot. Even with a hash guard
   suppressing no-change writes, real score arrivals rebuild
   everything — and the static topbar visibly blinks. Per-region
   wrappers (#regionTopbar / #regionOog / #regionStandings /
   #regionSidebar), each with display:contents and their own
   updateRegion hash guard, scope the rebuilds to the region that
   actually changed.

6. innerHTML REPLACE DESTROYS INNER SCROLL STATE
   When updateRegion writes new HTML into #regionStandings, the
   inner #standingsScroll element is replaced and its scrollTop
   defaults to 0. Auto-cycling animation pinned to the top forever.
   Fix: capture scrollTop before write, restore on the fresh
   element. updateRegion now does this for any region containing
   #standingsScroll.

7. DROPPING "(unconfigured)" IS A REAL UX FIX
   The fallback label looked broken on the public live page when
   the snapshot hadn't yet resolved a lens. Showing an empty sub
   line is better than parenthesized debug copy.

  feedback_no_keepalive_poll.md. WS-staleness detection should
  be message-timestamp-based.

  again. It's gone for a reason.
