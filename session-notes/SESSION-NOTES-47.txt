Spring II) with Engine v3.1.5. Session was driven by real-world bugs surfacing

  • Stale .cls test data leaking into Saratoga (engine pointed at a computer
    with a prior week's files in the watch folder)

tool, classes.schedule_order pipeline (matches Ryegate's tsked display
order), stats page mobile responsive, Scratched chip gates on FINAL, M4
live-box identity stacks 3 lines.

WORKER (west-worker.js)

  ── /scores-update lens fallback (e5dd58b) ──────────────────────────────
  Race fix: when /v3/postCls promotes a class from class_type='U' (initial
  unparsed state) to 'J/T/H', /scores-update was firing with snapshot.class_kind
  still null (stamped by /v3/postUdpEvent on the LAST UDP batch, before the
  .cls parse). The "no scoring lens" guard bailed without broadcasting → the
  first horse's standings never reached the page. Live page only caught up
  on the next UDP event (the second horse coming on course).

  Fix: derive lens from body.jumper_scores / body.hunter_scores when
  snapshot.class_kind is null. Also catch up snapshot.class_kind so
  subsequent reads stop seeing the stale null.

  ── /v3/flushRing endpoint (f49495c) ────────────────────────────────────
  Auth-gated POST { slug, ring_num }. Wipes for that ring:
    • D1: classes, entries, all entry_jumper_*/entry_hunter_*,
      class_jumper_stats, udp_events (filtered show_id+ring_num),
      ring_live_segment
    • R2: every object under ${slug}/${ring_num}/ (paginated)
    • KV: ring-state + cls-last (engine heartbeat preserved)
    • DO: routes through /class-action action=flush_all to nuke
      in-memory byClass + 15s cls_lock cooldown
  Show row + ring row preserved. Engine restart re-populates.
  Worker logs deletion summary.

  ── _focusedClassId pair-gating (5a9c9cd) ───────────────────────────────
  Per Bill: "the 31000 and the intro as a pair because this explicitly says
  this is the class we're on and this entry is going... it's very explicit."
  _focusedClassId now prefers the most-recently-locked is_live class (the
  one whose B+intro pair has fired). Falls back to eager Channel B / A only
  when no class has paired yet. Keeps the public panel from flapping when
  the operator browses classes by clicking around on Ryegate.

  ── _buildSnapshot idle-ring flush (9529692) ────────────────────────────
  When ringHasLiveClass is false (every class on the ring timed out / is
  FINAL / was cleared), null out the top-level last_identity / last_scoring
  / last_focus / class_meta / focused_class_id / focus_preview. Without
  this the M4 live box kept showing the last rider's name + horse + clock
  for hours after the ring went silent.

  ── _buildPrevEntry status detection (7709740) ──────────────────────────
  Now accepts r{N}_status as a round-attribution signal (was previously
  only checking total_faults / total_time / score_total). An eliminated
  rider has none of those — only the status field. _buildPrevEntry signature
  bumped to (row, classKind) so it knows whether to read r{N}_status (jumper)
  or r{N}_h_status (hunter). Returns status_code/label/category/full
  alongside existing fields. _samePrevEntry compares status_code so a
  clean→EL flip on the same entry/round still triggers the update.

  ── /v3/postTsked schedule_order write (4514766) ────────────────────────
  Migration 034 added classes.schedule_order INTEGER. Now /v3/postTsked
  writes idx+1 (1-indexed file position) for each tsked row. The public
  /v3/listClasses ORDER BY uses it (with class_id numeric fallback for
  unscheduled classes), so the public ring page matches Ryegate's display
  order — the operator's interleaved run sequence (325 / 925 / 930 / 330
  / 335 / 935...) instead of numeric sort.

  /v3/postCls catch-up also writes schedule_order — when a class first
  registers AFTER the operator's last tsked POST, the catch-up scans stored
  R2 tsked.csv tracking 1-indexed valid-row count.

  ── /v3/reprocessTsked endpoint (4514766) ───────────────────────────────
  Auth-gated backfill helper. POSTs slug as query param; reads tsked.csv
  from R2; re-runs the postTsked write logic. Used this session to backfill
  saratogaspring-wk2 (12 classes) and aiken-charity-2 (16 classes) without
  asking the operator to re-post tsked from Ryegate. Permanent admin tool.

NOTABLE BUGS HUNTED
═══════════════════════════════════════════════════════════════════════════

1. First-horse standings invisible on freshly-opened class
   D1 timeline: class registered with class_type='U' before .cls arrived
   → /v3/postUdpEvent stamped snapshot.class_kind=null → /v3/postCls parsed
   .cls (promoting to 'T') → /scores-update guarded on snapshot.class_kind
   and bailed → no broadcast. Page caught up only on next UDP. Fix derives
   lens from body's scores arrays.

2. Saratoga partial flush (operator + Claude both at fault)
   Engine was actively uploading while we tried a manual D1 wipe via MCP.
   Class 930 got recreated mid-flush with 5 entries from old test data.
   Hard lesson: operator must stop the engine first OR use the soon-to-ship
   /v3/flushRing button (which still requires engine stopped — the worker
   can't enforce the freeze; we just made recovery one click).

3. Chip says 711, name says "5-7 Year Old Young Jumper II,2b *YJC"
   Focused panel header chip from Channel A's last_scoring.class_id, name
   from Channel B's last_focus.tags[28]. When operator switches focus on
   Ryegate, Channel B fires immediately with the new class id+name, but
   Channel A doesn't fire until the next scoring event. Panel sat at "old
   number / new name" until the next scoring frame. Two-layer fix: page
   sources both fields from snapshot.focused_class_id + snapshot.class_meta
   (defensive); worker only commits focused_class_id when the B+intro pair
   has fired (preventive — eliminates the eager Channel B-only flap).

4. RIDER FALL latched on subsequent riders
   USER REPORT only — couldn't reproduce on clean state and the snapshot
   self-corrected within minutes. Suspect: lingering RF on Florence
   Jansen's r1_status (real fall earlier in the day) was bleeding through
   focus_preview because previous_entry got stale data with mismatched
   place/status. Filed as "watch item, retest after a clean flush" rather
   than chase further — tracked by a comment in west-worker.js for next
   time.

5. Just-finished banner blank when on-course rider was eliminated
   _buildPrevEntry only attributed a round when total_faults / total_time
   / score_total had data. Eliminated riders have ONLY r{N}_status='RF'/
   'EL'/'RT'/'WD' with everything else null. Detection missed them, returned
   null, banner stayed on the prior clean rider. Fix: status is now a valid
   round-attribution signal AND surfaces status_code/label/full on the
   returned record.

6. M4 live box stayed populated hours after ring went silent
   ringHasLiveClass=false (all classes timed out / FINAL'd) didn't clear
   the top-level last_identity / last_scoring / last_focus that carry
   through {...body} in _buildSnapshot. Fix: explicit null-out when no
   live class on ring; per-class previous_entry was already cleared on
   timeout.

7. Public ring page sorted classes alphabetically, not by tsked order
   Ryegate's tsked-driven order interleaves heights (325 / 925 / 930 / 330
   ...). Our /v3/listClasses ORDER BY only used scheduled_date + class_id,
   producing 325 / 330 / 335 / 340 ... / 925 / 930 / 935 / 940. Migration
   034 added classes.schedule_order; postTsked + postCls catch-up populate
   it; ORDER BY uses it; /v3/reprocessTsked backfills.

8. Scratched count inflated mid-class
   computeJumperStats counts entries with no round data — but mid-class
   that includes "haven't gone yet" riders. Stats page now hides the
   Scratched chip until cls.finalized_at is set.
