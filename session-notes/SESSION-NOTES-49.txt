Continuation of S48 in-show day (Aiken Charity Show II + Saratoga Spring
II). Started with small banner polish ("show all rounds + Overall when
N>1") and ended with a complete rebuild of the just-finished banner
into a worker-driven slot list.

22 commits after S48 was cut. Three big arcs:

1. Hunter banner correctness:
   • Round breakdown when class is past round 1
   • Trim trailing zeros (86 not 86.0; 85.5 not 85.50; 85.75 stays)
   • Status-aware rendering — eliminations render "Status EL" or
     "R2 RF · R1 86" instead of treating r2_score_total=0 as a
     real score
   • previous_entry promote logic: hunted multiple race conditions
     between UDP events and /v3/postCls row writes

2. Banner architecture rebuild (Bill: "this shouldn't have been
   this hard keep it simple"):
   • Worker emits pe.banner_slots = [{label, value, emphasize?}, ...]
   • Page is a 5-line iterator
   • Killed ~80 lines of branching client-side logic
   • Dragged out an esc-not-defined ReferenceError that was silently
     killing the multi-round path on mobile

3. Class header pill + 'X of Y' progress (designed for kiosk reuse):
   • Worker emits snapshot.pill + class.pill
   • {state, label, progress, progress_label, total, remaining, gone}
   • Pages render directly — single source of truth across surfaces
   • Stack: pill on top, value '27 of 40' centered below, 'GONE'
     caption underneath
   • FINAL state suppresses progress

Plus: cache-bust the preview pipeline so mobile Safari can't serve
stale bundles after a deploy. Bill confirmed it kept hitting cached
old code despite hard refreshes.

WORKER (west-worker.js)

  ── Hunter banner shape detection (8d5beb6, 1eafefe) ─────────────────────
  • _buildPrevEntry now emits pe.rounds = [{n, label, score}] for
    every round with non-null r{N}_score_total + no terminating status.
  • _decodeHunterDisplayedRound subset-matches combined_total against
    the released-rounds set so the banner can show "Overall 173.5"
    when the operator hits Display Overall and "R1 86" when only
    R1 was released.
  • _samePrevEntry compares combined_total + displayed_round_label
    in addition to entry/round/faults so an Overall release on the
    same entry re-promotes.

  ── Status-aware rendering (dd9fa8f) ──────────────────────────────────────
  Subset-match algorithm on BOTH worker (_decodeHunterDisplayedRound)
  and template (renderLowerThird) used to treat r{N}_score_total=0
  (the convention for status-set rounds) as a real zero score and
  falsely flipped to Overall. Both paths now skip rounds with
  r{N}_h_status set.
  M4 lower-third also short-circuits when the highest round has a
  terminating status — renders "R2 RF · R1 86" or "Status EL" when
  no clean prior chips exist.

  ── Hunter score formatter (d924813) ──────────────────────────────────────
  Added WEST.format.hunterScore(v) — 2 decimals max, trailing zeros
  stripped (86 → "86", 85.5 → "85.5", 85.75 → "85.75"). Replaced
  every Number(x).toFixed(1) callsite in the M4 lower-third + judge
  cells + score card. Each function also has a local fmt fallback so
  a stale cached west-format.js doesn't kill rendering.

  ── pe-promote race-condition saga (0e7612c → 28ae9b8) ───────────────────
  Hunted four different races over five commits before landing on
  the right architecture:

  1. (0e7612c) Don't promote the INCOMING on-course rider when their
     row carries prior-round data from earlier in the class.
  2. (10a0afe) Round-2 fix: in case (a), only promote if the row
     CHANGED since the prior batch (compare candidate to prior-row
     candidate via _samePrevEntry).
  3. (38cd4af) Add judges_sig fingerprint to _samePrevEntry so
     late-arriving judges (which land after /v3/postCls writes the
     base row) re-promote with J1/J2 cells.
  4. (718cb26) Defer pe promote until row data settles — case (a)
     [same rider] handled exclusively in /scores-update.
  5. (28ae9b8) Final fix: kill case (b) too. /scores-update is the
     SOLE pe promote site. It detects the just-finished rider via
     signature-diff (compare body.hunter_scores vs existing.hunter_scores
     for the entry whose row data CHANGED) — doesn't depend on
     last_identity, which can flip to next rider before /scores-update
     fires.

  Trade-off: brief gap on rider transitions (a few hundred ms) where
  banner stays on prior promoted entry until /v3/postCls completes.
  Single clean update wins over snappier-but-flashy.

  ── banner_slots architecture (a6e4205) ──────────────────────────────────
  Worker now decides the entire banner shape:

    EL/RF/RT/WD: Status / Reason / Rank
    Hunter multi-round (>=2 rounds released):
      R1 / R2 / [R3] / Overall(emph) / J1 / J2 / Rank
    Hunter single-round: Score(emph) / J1 / J2 / Rank
    Jumper / equitation: F / Time / Rank

  Single-judge classes skip judge slots. Derby classes inline hi-opt
  + handy on judge slots ("J1 88+5+3"). Page is a 5-line iterator.
  Replaced ~80 lines of client-side branching.

  ── Per-judge slots + judges_sig (d9222de, 38cd4af) ──────────────────────
  Multi-judge classes get J1, J2, ... slots from the LATEST released
  round's judges. Worker emits judges_sig (round:idx:base:hiopt:handy
  fingerprint) in pe so _samePrevEntry detects when judge data
  arrives after the initial promote and re-builds banner_slots.

  ── Class pill + remaining-progress (c6f6a13, c702f7c, ed0d658) ──────────
  _buildClassPill(classEntry) emits:
    { state: 'final'|'inring'|'open',
      label: 'FINAL'|'In Ring'|'Open',
      progress: '27 of 40' | null,
      progress_label: 'Gone' | null,
      total, remaining, gone }

  Attached to each class in snapshot.classes[] AND to top-level
  snapshot.pill (focused class). Page renders pill on top, value
  centered below, 'GONE' caption underneath. FINAL suppresses
  progress entirely.

  Initial direction was "remaining of total" with 'Remaining' label,
  Bill flipped it to count UP (gone of total) because the number
  ticking down read backwards as the class progressed.

  ── pe judges field exposed (38cd4af) ────────────────────────────────────
  pe carries `judges` array directly so the page-side fallback
  (computeSlotsFromPe) can build judge slots if banner_slots is
  somehow missing.

PAGES — live.html

  ── Banner DOM + render simplified (a6e4205, 822d327) ────────────────────
  Render code dropped from ~80 lines of conditional branches to a
  single iterator:

    jfStatsRow.innerHTML = pe.banner_slots.map(s =>
      `<span class="jf-stat${s.emphasize ? ' is-emphasis' : ''}">
         <span class="jf-stat-lbl">${jfEsc(s.label)}</span>
         <span>${jfEsc(s.value)}</span>
       </span>`
    ).join('');

  THE BUG (822d327): the previous render code referenced `esc`, which
  was declared inside renderOtherClassPanels(), not inside render().
  At banner render time `esc` was undefined → ReferenceError → inner
  block bailed silently → only the identity ($('jfNum')/jfName) made
  it to the DOM, stats stayed as static placeholder HTML. Mobile
  Safari was the canary here because Chrome MCP preview happened to
  evaluate paths in a slightly different order that masked the
  failure.

  Lesson logged: when adding new code blocks inside render(), declare
  ALL helpers (esc, fmt, etc.) locally in scope. Don't borrow from
  other functions' closures.

  ── computeSlotsFromPe page-side fallback ────────────────────────────────
  Mirrors the worker's banner_slots logic so pe records that
  pre-date the worker emitting banner_slots still render. Once every
  in-flight pe carries banner_slots, the fallback never fires.

  ── Mobile breakpoint: J1/J2 wrap to row 2 (219bb79) ─────────────────────
  Round + Overall on row 1 (the headline numbers), judges + Rank wrap
  to row 2 below 640px. Implemented via a sibling `.jf-break` element
  with flex-basis:100% inserted before the first J{n} slot. Desktop
  unchanged — break element is display:none above 640px.

  ── Score-summary slot emphasized (7cc929d) ──────────────────────────────
  Multi-round Overall and single-round Score render at 16px instead
  of 12px so the eye lands on the headline number first. Worker emits
  emphasize:true on those slots; page adds .is-emphasis class. Fallback
  also detects label name ('Overall' / 'Score') for backward-compat.

PAGES — class header pill

  ── Two-row stack pill + 'X of Y' + 'Gone' caption ───────────────────────
  Header status column is now flex-column, center-aligned. Pill on

DEPLOY PIPELINE

  ── Cache-bust per preview deploy (be82277) ──────────────────────────────

PRINCIPLES REINFORCED
═══════════════════════════════════════════════════════════════════════════

1. WORKER DECIDES SHAPE, PAGE ITERATES
   pe.banner_slots and class.pill are both worker-emitted descriptors
   the page renders directly. Adding new slot types or pill states is
   a worker-only change. live.html, ring display, future scoreboard
   all read the same fields.

2. WAIT FOR ROW DATA TO SETTLE
   pe is promoted ONLY by /scores-update (the path called after
   /v3/postCls writes fresh row data to D1). _updateByClass no longer
   touches pe. Single visible banner update per operator press, no
   flash-then-correct.

3. SIGNATURE-DIFF FOR "WHO JUST FINISHED"
   /scores-update picks the promote target by diffing scoring fields
   between body and existing scores arrays — not by reading
   last_identity (which can flip to next rider before
   /scores-update fires).

4. KEEP HELPERS LOCAL TO WHERE THEY'RE USED
   esc-scope bug burned ~30 minutes of debugging because esc was
   declared in a different function's closure. Adding code inside
   render()? Declare all helpers in render().
