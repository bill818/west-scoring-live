PHASE 2c — entries parsing + admin drill-down
  - Migration 004: entries table (class_id FK, entry_num, horse,
    rider, owner, USEF nums, city, state, raw_row, timestamps).
    UNIQUE(class_id, entry_num).
  - parseClsEntriesV3 — identity-only (cols 0-12, shared across
    H/J/T). Skips @metadata rows. Validates entry_num.
  - Lens rule (Bill's call): parse entries when lens is known.
    H/J/T always. U WITH inferred method → jumper lens. Bare U
    skipped silently.
  - GET /v3/listEntries?class_id=N
  - Admin class rows are click-to-expand → entry table (horse,
    rider, owner, USEF, city/state).
  - Per-class stale-entry cleanup: on re-POST, entries not in the
    current .cls get deleted (operator-removed entries disappear).

PHASE 2c.5 — tsked parsing + day-grouping
  - Migration 005: classes.scheduled_date TEXT, classes.schedule_flag
    TEXT.
  - POST /v3/postTsked — archives raw to R2 at {slug}/tsked.csv,
    parses rows, UPDATE-only (never creates classes — .cls POST is
    authority). Normalizes M/D/YYYY → ISO YYYY-MM-DD.
  - Clear-stale: classes previously scheduled but no longer in tsked
    have scheduled_date + schedule_flag nulled (they fall to
    "Unscheduled" bucket in admin — no data loss).
  - Engine watches C:\Ryegate\Jumper\tsked.csv with SHA-256
    content-hash check (Session 25 "Upload Results mtime no-op touch"
    fix).
  - Admin: two-level grouping Ring → Day → Classes with "Fri
    MM/DD/YYYY" day headers. "Unscheduled" bucket at bottom of each
    ring.
  - Schedule-flag badge rendered next to class name (S=Scored green,
    JO=JO-posted blue, L=Live amber).

PHASE 2c.6 — soft-delete tracking + download + hard-delete
  - Migration 006: classes.deleted_at TEXT.
  - Engine: unified scheduleClsEvent(). On fs.watch fire, check if
    file exists → POST or DELETE accordingly. Rapid delete+restore
    sequences collapse correctly via debounce.
  - POST /v3/deleteCls — sets deleted_at on matching class. Guarded
    WHERE deleted_at IS NULL (idempotent).
  - POST /v3/postCls now clears deleted_at on re-POST. Drop the file
    back → class auto-undeletes.
  - GET /v3/downloadCls?slug=X&ring=N&class=C — streams raw R2 bytes
    back as a file download. Admin fetches via JS + Blob + anchor
    click so auth stays private.
  - POST /v3/hardDeleteCls — deletes entries + class row + R2 object.
    Two-step safety: only works if class is already soft-deleted
    (returns 409 otherwise).
  - Admin "Deleted from Ryegate" section at the bottom of Classes.
    Rows: strikethrough name, [⬇ Download] + [🗑 Delete Forever]
    buttons. Confirm dialog before hard-delete.
  - Smoke-tested full round-trip: create 999.cls → upload → delete
    file → soft-delete triggered → restore file → auto-undelete.
    Hard-delete also verified.

PHASE 2c.7 — v2 show fields + unified Add/Edit dialogs

  - Show header displays: slug, dates, venue, location, status,

ARCHITECTURAL RULES LOCKED THIS SESSION

1. SLUG RENAME PERMANENTLY OUT OF SCOPE. Bill: "slug rename introduces
   errors i don't like." Future "migrate show to new slug" operation
   is theoretically possible (clone + transfer R2 + class links + old
   delete) but not planned. /v3/updateShow does not accept slug.

2. RING_NUM IMMUTABLE on existing rings — same rationale as slug.

3. ADMIN MIRRORS RYEGATE FOLDER. Classes that are soft-deleted
   stay visible in admin's "Deleted from Ryegate" section with
   ⬇ Download + 🗑 Delete Forever actions. No silent data loss.

4. PUBLIC-FACING vs ADMIN-FACING GROUPING DIFFER.
   - Admin sidebar: status-driven (Active = status='active' etc.)
   - Public index: date-driven (Happening now = today between start/end)
   Operator cares about "what state did I set"; spectator cares about
   "what's happening today". Different audiences, different models.

5. SHOW STATUS AUTO-LIFECYCLE.
   - pending (default on create)
   - heartbeat arrives → worker auto-flips pending → active
   - Manual changes to complete / archived are sticky — engine
     activity doesn't demote.

6. OPTION B R2 ARCHIVE: one .cls per class, overwrite-in-place. No
   versioned history. Raw tsked also overwrite-in-place. History
   snapshots can be added later if needed.

7. UNIFIED DIALOG PATTERN FOR ADD/EDIT. Shows and Rings both use one
   dialog each with mode toggle ('create' | 'edit'). Primary key
   locked in edit mode. Consistent UX, one code path per entity.

8. FLAG CENTRALIZATION: all method/mode/modifier/schedule-flag labels
   live in v3/js/west-format.js. Admin pulls via WEST.format.* —
   hunter modes are currently empty map {} (Bill hasn't audited
   Ryegate settings yet) and fall back to "Hunter classMode N".

9. ENTRY COLUMN MAP IS INLINE IN WORKER (Phase 2c). Not yet in
   west-rules.js because only one consumer (worker). Extract when
   a second parser needs it.

10. ENGINE NEVER POSTS DUPLICATE CONTENT for tsked.csv. SHA-256
    content-hash check before POST. Session 25's "Upload Results
    no-op mtime-touch" trap avoided.
