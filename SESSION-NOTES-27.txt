ACT 1 — THE REWRITE THAT FAILED (mid-show)
------------------------------------------
The on-course ticking clock (#woc-clock on live.html) had a
long-standing visible flicker: render() and tickOnCourse()
both wrote to the same DOM element every poll with slightly
different interpolated values, ~1s apart, using mismatched
timestamps (worker time vs browser time).

Morning attempt: full heartbeat-as-authority rewrite of
tickOnCourse/setOcState/applyHeartbeatClock. It went sideways:

  1. render() replaces innerHTML every poll — destroyed
     tick's written number between frames
  2. Heartbeat applied without entry-match → flashed the
     previous horse's clock after a horse swap
  3. Heartbeat stomped the whole _ocState object instead of
     just adjusting interpolation base
  4. FINISH — tick overwrote render's real decimal ("78.001")
     with integer from heartbeat ("78.000")
  5. Null state between horses never cleared the DOM

Each fix made something else worse. Classic mid-show trap.
Around noon Bill called it: "this clock is really broken.. it's
wrong more than it's right at this point."

RESOLUTION: git reset display-config.js and live.html to HEAD,
kept ONE line — WEST.tickOnCourse() called synchronously after
render() in the poll loop. Same JS frame, no paint between,
tick's value is the only thing the user sees.

That one-line fix shipped as commit 4d45066, pushed before
dinner. Show finished on reverted code with visible-flicker-
but-accurate behavior.


ACT 2 — LOG-DRIVEN REDESIGN (late night)
-----------------------------------------
Bill pulled the three log files from the scoring PC's
OneDrive folder and asked for an evaluation:

  west_log.txt       (257 KB, watcher activity)
  west_peek_log.txt  (76 KB, classifier)
  west_udp_log.txt   (3.0 MB, raw UDP feed — 30k lines)

THE KEY FINDING: The UDP source is a clean 1-Hz stream.

  CD:       -45, -44, -43, -42... every 1.00s
  ONCOURSE: el=1, el=2, el=3... every 1.00s
  FINISH:   el=93 → el=93.367 in the same second
  RANK:     arrives as separate frame ~5s after FINISH

Zero missed frames across hours of riding. Zero jitter.

This completely invalidates the original architectural
assumption. The "timestamp + browser-side interpolation"
design was built to avoid "jerky 1,4,7,10 UDP counting" —
but the UDP doesn't do that. It never did. The source is
already a perfect clock, and every design decision that
flowed from the jerky-UDP assumption was solving a
non-problem while creating real ones.

Second finding: heartbeat was firing but invisible. Only
one "Heartbeat sent to Worker" log line at startup, because
postToWorker is silent on success. Actual 10s interval was
firing the whole time, nobody could tell from logs whether
the worker was receiving them.

Third finding: everything else in the watcher is healthy.
tsked/peek classifier clean, phase transitions clean, no
buzzer noise in the log, FINISH→RANK two-phase working.


THE v1.11 DESIGN (shipped)
--------------------------
Heartbeat is the authority. Every 1s it carries the full
clock snapshot {phase, elapsed, countdown, entry, ta,
jumpFaults, rank}. Browser trusts the heartbeat absolutely.
Local tick runs at 1Hz only to fill sub-second visual gaps
between polls — it never extrapolates from cross-clock
timestamps. When a heartbeat arrives, local tick snaps to
the authoritative value and resumes.

The analogy Bill liked: "the watcher shouts the number once
a second — 12!, 13!, 14! — the browser also counts quietly
between shouts just to fill the silence. If a shout is
missed, no one notices. If two arrive fast, browser just
resyncs silently."

    → tickOnCourse, all in same JS frame, single paint

That last line reset the weight of the whole conversation. This
isn't a cleanup project — it's business-critical infrastructure
for Bill's company, visible to paying show managers and hundreds
of spectators. Failure is a credibility hit on a live stage.

New memory file: project_stakes.md, pinned at top of MEMORY.md.
Rules that fall out:
  - Never ship unproven code mid-show (S27's lesson, permanent)
  - Reliability > cleverness when tied on correctness
  - Every rollback path is a feature, not overkill
  - Shadow-run the WebSocket rebuild, migrate page by page
  - Treat reliability concerns as the signal they are

WebSocket/DO rebuild promoted from "future" → "ACTIVE,
starting April/May 2026" in memory. Approach: consolidate
shared logic FIRST (the audit), then layer push architecture
on top, page by page, with polling as fallback during
migration. Bill wants Devon co-leading.
