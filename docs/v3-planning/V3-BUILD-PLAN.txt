HOW TO READ THIS DOC
--------------------
This is a DOING doc, not a THINKING doc. For the thinking
see: WEBSOCKETS-OVERVIEW.txt, CENTRALIZED-JS-ARCHITECTURE.txt,
STATS-MODULE-ADDENDUM.txt.

Structure:
  - 11 phases, sequenced
  - Each phase: goal, files touched, concrete steps,
    acceptance criteria ("how you know it's done"),
    rollback plan, estimated sessions
  - Don't start phase N+1 until phase N meets its
    acceptance criteria
  - Sessions are 2-4 hour focused blocks, not calendar days

TOTAL SCOPE ESTIMATE: 25-35 sessions over 2-3 months of
calm work between shows. Build realistic expectations —
skipping phases or rushing merges is how this dies.


============================================================
  VERSIONING AND BRANCHING STRATEGY
============================================================

VERSION SCHEME:
  Watcher: continues v1.x (independent, field-deployed)
  Worker + frontend: jump from current (call it v2.x) to
    v3.0.0 when all phases are complete and cut over.
  During the rebuild, incremental tags: v2.5.0, v2.6.0, etc.,
    one per phase that ships to production.

BRANCH MODEL:
  main — always shippable, always what's in production
  v3-rebuild — the rebuild track. NEVER deploys to prod
    directly; only merges back to main when a phase is
    stable and verified in preview.
  Short-lived feature branches per phase come off v3-rebuild.

SAFETY RULE (HARD):
  Never merge v3-rebuild → main during a show weekend.
  Only merge to main on weekdays with a show-free cushion
  of at least 3 days.


============================================================
  PHASE 0 — PREPARATION (1-2 SESSIONS)
============================================================

GOAL: Create the working environment, baselines, and
safety nets before writing any rebuild code.

STEPS:

0.1 Tag current production as v2.x baseline
    git tag v2.9.0-pre-rebuild
    git push origin v2.9.0-pre-rebuild
    This is the "known good" rollback point for the entire
    rebuild effort.

0.2 Create long-lived rebuild branch
    git checkout -b v3-rebuild
    git push -u origin v3-rebuild

0.3 Set up a /js/ folder for the new modules
    Empty placeholder files so import paths work from day 1:
      /js/west-format.js
      /js/west-rules.js
      /js/west-clock.js
      /js/west-display.js
      /js/west-stats.js
      /js/west-data.js
    Each file starts with just:
      (function() { window.WEST = window.WEST || {}; })();

0.4 Add a v3-flag in the worker
    A feature flag in west-worker.js so new endpoints can
    be shipped behind it without affecting production.
    const V3_ENABLED = url.searchParams.has('v3') ||
                       env.V3_ENABLED === 'true';
    This lets preview + staging test v3 paths while main
    stays on v2.

0.5 Set up a /tests/ folder + minimal test runner
    Vitest is fastest to set up:
      npm init -y
      npm install --save-dev vitest
      create /tests/sample.test.js that does
        import { test, expect } from 'vitest'
        test('sanity', () => expect(1+1).toBe(2))
      Add to package.json:
        "scripts": { "test": "vitest" }
    Run: npm test — verify it works.

0.6 Document the contract of each existing HTML page
    Write a 1-page-each summary of what live.html, display.html,
    stats.html, classes.html, index.html currently do, what
    data they need, what the user can do on them. These
    become the acceptance criteria when we rebuild them —
    "does the new page still do everything this list says?"
    Save to /docs/page-contracts/ in the repo.

0.7 Create a realistic replay harness
    Copy today's UDP log into /tests/fixtures/culpeper-day-1.log
    Write a tiny Node script that can replay the log against
    a local watcher or mock watcher, producing the same
    sequence of /heartbeat, /postClassData, etc. POSTs.
    This is how we test the rebuild without needing a real
    show running.

ACCEPTANCE CRITERIA:
  [ ] v2.9.0-pre-rebuild tag exists on origin
  [ ] v3-rebuild branch exists on origin
  [ ] /js/ folder contains six empty stub files
  [ ] npm test runs sample.test.js successfully
  [ ] /docs/page-contracts/ has one file per page
  [ ] Replay harness can produce a POST sequence from log

============================================================
  PHASE 3 — WEST-CLOCK.JS (2-3 SESSIONS)
============================================================

GOAL: One clock implementation. Kill the three
duplicates in live.html, display.html, stats.html.

FILES TOUCHED:
  + /js/west-clock.js
  + /tests/west-clock.test.js
  ~ live.html (migrate from display-config.js clock helpers)
  ~ display.html (delete local tick)
  ~ stats.html (delete local tickClock)
  ~ display-config.js (move clock helpers out, clean up)

STEPS:

3.1 Create /js/west-clock.js with the v1.11-proven model:
      WEST.clock.setState(state)
        state = { phase, elapsed, countdown, ta, entry, paused }
      WEST.clock.tick()
        Called on setInterval. Reads state, writes DOM.
      WEST.clock.bind(element, opts)
        opts = { precision, showPhase, showFaults }
      WEST.clock.applyAuthoritative(heartbeat)
        Replaces current applyHeartbeatClock.

3.2 Port the v1.11 logic from display-config.js verbatim.
    This phase is NOT redesigning the clock — just moving
    the working v1.11 code into its own module.

3.3 Write tests. Clock tests are fiddly because of Date.now()
    and setInterval. Use fake timers (Vitest has built-in
    support). Test cases:
    - setState → tick writes correct value
    - Authoritative update resets interpolation base
    - Phase-priority guard blocks regression
    - Entry-match guard rejects stale entry data
    - FINISH phase doesn't tick

3.4 Migrate live.html first. Replace calls to
    display-config.js clock helpers with WEST.clock.*.
    The behavior should be IDENTICAL to today — we're just
    relocating code.

3.5 Migrate display.html. Its old tick() function had
    100ms interval and different element IDs. Use
    WEST.clock.bind('#liveClock', { ...opts }) to point
    the shared clock at its DOM. Delete the local tick().

3.6 Migrate stats.html same way. Delete local tickClock().

3.7 Delete the clock helpers from display-config.js.
    Anything they did that wasn't clock-related already
    lives elsewhere.

ACCEPTANCE CRITERIA:
  [ ] /js/west-clock.js is the ONLY place a ticking
      clock is computed
  [ ] live.html / display.html / stats.html all use it
  [ ] Their local clock functions are DELETED, not
      commented out
  [ ] west-clock.test.js passes
  [ ] Preview: clock behavior on all three pages matches
      (or consistently improves on) production
  [ ] Tested against replay harness with Day 1 UDP log

MERGE BACK TO MAIN: Yes, tag v2.12.0. This is the first
phase where the "consolidation debt" is paid off for a
specific subsystem.

ROLLBACK: Per-page reverts. Clock rollback in particular
needs careful verification since it's the most-used
feature on a live page.

============================================================
  ADDENDUM A — .CLS AS SOURCE OF TRUTH
  (NO MORE TRIAL-AND-ERROR)
============================================================

Bill's requirement: "I dont ever want to sit there and trial
and error every class settings we need to take what we
learned and apply it so we can streamline it." — clarified:
"these class settings need to be read from the .cls files."

THE KEY INSIGHT:
The .cls file IS the source of truth. Farmtek writes it.
Ryegate writes it. It describes the class — scoring method,
time allowed, round structure, flags, everything. Our job
is to READ it correctly, not to reinvent it in a hardcoded
profile table.

What trial-and-error usually comes from isn't "what should
this class DO" — Farmtek already decided that. It's "which
column in the .cls file holds this piece of information on
this particular flavor of class." Ryegate shifts columns
between class types. Numeric status codes mean different
things in different rounds. Field meanings change based on
scoring method. Those are PARSER quirks, not behavior choices.

THE SOLUTION IN v3:
Build ONE canonical .cls parser that understands every
quirk of every class type Farmtek emits. The parser output
is a complete, normalized CLASS DESCRIPTOR that every page
can use without further interpretation.

THE PARSER LIVES WHERE?
On the worker, not the frontend. The frontend never sees
raw .cls data. The worker parses .cls into a normalized
descriptor, stores it in KV/D1, and serves it via
/getLiveClass. Pages consume the normalized shape.

west-rules.js on the frontend still exists — it helps pages
INTERPRET the normalized descriptor for display purposes
(e.g. "given this descriptor, should the row show jump
faults or a hunter score?"). But it never touches .cls
columns directly.

NORMALIZED CLASS DESCRIPTOR SHAPE (what the parser outputs):
  {
    classNum: '296',
    className: '$200 6 Year Old Jumper',

    // Direct from .cls — canonical, no interpretation
    scoringMethod: '2',
    scoringMethodLabel: 'Jumper — Time First',
    classType: 'J',
    timeAllowed: 90,
    faultsPerInterval: 1,
    timeInterval: 1,
    roundCount: 2,
    rounds: [
      { index: 1, label: 'Round 1', timeAllowed: 90 },
      { index: 2, label: 'Jump Off', timeAllowed: 55 }
    ],

    // Flags from .cls — explicit, named, not magic columns
    flags: {
      isEquitation: false,
      isOptimum: false,
      isTwoPhase: false,
      isHandy: false,
      isFaultsConverted: false
    },

    // Derived fields the parser computes so pages don't have to
    derived: {
      optimumTime: null,
      phase1TA: null,
      phase2TA: null,
      primaryIdentity: 'horse',  // vs 'rider' for eq
      sortKey: 'time-then-rank'
    },

    // Parser metadata — which .cls version, any quirks
    meta: {
      clsVersion: 'Farmtek 4.2',
      parseWarnings: []
    }
  }

THE KNOWLEDGE WE CODIFY:
Not "what method 7 does" — that's already decided by Farmtek.
What WE codify is the PARSER's knowledge of Farmtek's quirks:

  - "When scoringMethod is 7, read score from column 28
     not column 21"
  - "Numeric status codes in column 35 mean X for first
     round, Y for jump-off"
  - "When classType is H and hasEqFlag is true, treat
     scoringMethod 7 as pinned equitation"
  - "Two-phase classes (method 9) write time as TA:55 in
     phase 2 even though phase 1 used TA:90"
  - "Round labels come from cls[H12] not cls[H09]"
  - [... every Ryegate quirk Bill has ever hit ...]

Each quirk gets a NAMED, COMMENTED block in the parser
with the real example that taught us the quirk.

EXAMPLE PARSER SNIPPET (conceptual):
  function parseCls(raw) {
    const descriptor = { ... };

    // Quirk: Ryegate shifts the numeric status column
    // between cols 21/28/35 depending on which round is
    // active. Learned 2026-04-16 from Culpeper Day 2 when
    // JO WDs weren't appearing in standings.
    descriptor.derived.statusColumn =
      raw.scoringMethod === '9' && raw.activeRound === 2
        ? 35
        : raw.classType === 'H' ? 28 : 21;

    // Quirk: Method 6 optimum classes compute optimum
    // as TA minus 4 seconds, per FEI/USEF rules. Not in
    // .cls directly.
    if (raw.scoringMethod === '6' && raw.timeAllowed > 0) {
      descriptor.derived.optimumTime = raw.timeAllowed - 4;
    }

    // ... every quirk codified with source and date
    return descriptor;
  }

HOW WE FIND QUIRKS WE HAVEN'T MET YET:
Same pattern as the old "profile library" idea:
  1. Parser runs against every .cls
  2. Fields that can't be interpreted log a parse warning
     with full context (class metadata, raw .cls dump)
  3. Bill reviews warnings after the show
  4. Parser gains a new named quirk-handler
  5. Next class of that type is handled automatically

Parse warnings go into the stats_rebuild_log table (or its
cousin) so they're queryable, not just ephemeral console logs.

WHAT STAYS IN west-rules.js (FRONTEND):
The frontend rules module is thin. It takes the normalized
descriptor (already parsed) and helps pages decide layout:
  WEST.rules.rowLayout(descriptor) → 'jumper' | 'hunter' | 'eq' | ...
  WEST.rules.showsTimeFaults(descriptor) → bool
  WEST.rules.roundLabels(descriptor) → ['Round 1', 'Jump Off']
  WEST.rules.statusSort(descriptor) → sort function

These are small wrapper functions that READ the descriptor's
fields — they don't re-parse or re-derive. If a rule gets
complicated, push the derivation back into the parser's
"derived" block so it's done once per class, not once per
spectator render.

BUILD PLAN IMPACT:
Phase 2 refocuses:
  - PRIMARY deliverable: .cls parser on the worker with
    the quirk catalog
  - SECONDARY deliverable: west-rules.js frontend wrappers
    that read the normalized descriptor

Phase 2 session estimate: 3-5 sessions (same as before,
split differently).

WATCHER IMPLICATION:
The watcher already parses .cls (it reads the file and
extracts metadata). It currently posts /postClassData with
whatever it extracts. The v3 plan moves the canonical
parser to the WORKER so the browser and worker agree on
interpretation.

Short-term (during rebuild): watcher continues posting as
today. Worker re-parses server-side to produce normalized
descriptor. Eventually watcher could post raw .cls content
to the worker and let it be the only parser — but that's
optimization, not a rebuild blocker.

TEST DISCIPLINE:
Every quirk in the parser has a test case with a REAL
.cls fixture that demonstrated the quirk. When Ryegate
changes format (they will), the fixture test catches it.

Quirks without fixtures = quirks we don't fully understand.
Don't ship those — log the unknown case, gather real data,
then write the handler with its real fixture.

THE LONG VIEW:
In 2 years, the parser should handle every .cls variation
west-scoring-live has seen. New shows become zero-config.
An operator setting up a show never opens a "class profile
editor" because there isn't one. The .cls says what the
class is. The parser knows what that means. The frontend
just displays it.

If a new Farmtek version ships with a new field, the
parser catches it as a parse warning, Bill reviews it,
codifies it as a named quirk. No trial and error.

DISCIPLINE RULE:
No page, no module, no worker endpoint reads raw .cls
columns except the parser. If you ever find yourself
doing raw[21] or similar outside parseCls(), that logic
belongs IN the parser, as a named quirk with a comment.


============================================================
  ADDENDUM B — STATS DATABASE DECISION
============================================================

Bill's question: "do we need a secondary stats database or
should that fold into the existing database?"

THE ANSWER: Same D1 database, separate tables. Not a second DB.

WHY ONE DATABASE:

1. JOINS ACROSS ENTRIES + STATS
   The rollup process needs to walk raw entries to
   recompute stats. That's a query like:
     SELECT rider_id, COUNT(*), SUM(earnings)
     FROM entries
     JOIN results ON ...
     GROUP BY rider_id
   Cross-database JOINs in D1 don't work cleanly — SQLite's
   ATTACH isn't available in Workers' D1. Putting entries
   and rollups in the same DB keeps queries simple.

2. ONE BACKUP STORY
   D1 backups are per-database. One database = one backup
   procedure, one restore procedure, one retention policy.
   Two databases doubles the operations story with no
   operational benefit.

3. ONE MIGRATION STORY
   Schema changes happen in one place. When you add a
   column to entries, you don't have to coordinate with a
   separate stats DB's schema.

4. WRITE CONTENTION ISN'T A REAL PROBLEM
   Stats tables are read-mostly. Rollup cron writes to
   them, and client requests only read. Entry writes go to
   the entries tables. Different table = no contention.
   You'd have the same isolation benefit without the
   overhead of a second database.

5. COST
   D1 pricing is per-request + per-row. Two databases
   means two billing entries. Not a big deal but it's
   strictly worse than one.

WHEN YOU WOULD WANT A SECOND DATABASE:
  - If stats writes were so heavy they contended with
    entries writes (they won't — rollups happen nightly/
    15-min during shows, not per-entry)
  - If stats had different durability requirements (they
    don't — both are important)
  - If you wanted per-database access controls (D1 doesn't
    really support this at the table level anyway)

None of these apply to west-scoring-live.

TABLE STRUCTURE IN THE ONE DB:

Existing (roughly — don't modify schema during rebuild):
  entries, results, classes, shows, ...

New tables for stats (added in Phase 9):
  rider_aliases            — identity reconciliation
  rider_season_stats       — rollup per rider per season
  horse_season_stats       — rollup per horse per season
  show_summary_stats       — per-show rollup
  class_leaderboard        — per class-type season
  stats_rebuild_log        — audit of rollup runs

Name them with a prefix or suffix (stats_* maybe) so
when you're looking at the schema it's obvious which
tables are raw-data vs rollup.

BUILD PLAN IMPACT:
Phase 9 (Historical Stats Engine) stays in one database.
D1 migrations add the stats_* tables. No second D1
binding needed in wrangler.toml. One environment
variable (WEST_DB) continues to be the only DB reference.

ONE GOTCHA WORTH NOTING:
D1 has per-database size limits (currently generous but
not infinite — check current limits when you get to Phase 9).
If historical data across 5+ seasons grows too large, the
scaling answer isn't "split DBs" — it's archive cold data
to R2 and keep only the last 2 seasons in D1.

If you ever hit that wall, it's a good problem to have.
Cross that bridge in 2028, not 2026.
